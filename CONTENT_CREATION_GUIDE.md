# KLT 콘텐츠 생성 자동화 지침 (CONTENT_CREATION_GUIDE)

> **이 문서는 AI 모델 공통 지침이다.** 어떤 모델이 작업하든 이 지침과 `contents/_templates/`의
> 템플릿을 그대로 따르면 동일한 구조·톤·퀄리티의 콘텐츠가 나와야 한다.
> 원천 규격: `2026_영업파이프라인다각화_실행가이드` (4-4 AI 콘텐츠 제작 프롬프트 가이드)

---

## 1. 시스템 개요

```
[작업자] "주제는 ○○○. 내용은 ○○○. 컨텐츠 생성 시작."
   ↓
[AI] ID 부여 → contents/{ID}_{슬러그}/ 폴더 생성 → 01_main.html (마스터 원고) 생성
     → Supabase content_master 등록 (검수상태 '대기')
   ↓
[작업자] 검수. 수정 필요 시 "반려: {사유}" → AI가 01_main.html 수정 후 재검수
   ↓
[작업자] "발행 승인"
   ↓
[AI] 검수상태 '승인' + 발행일(published_date) 자동 기록
     → 02~07 파생 콘텐츠 5종 순차 생성 → 파일 생성 플래그 Supabase 기록
   ↓
[작업자] 각 채널에 복사·붙여넣기 게시 → 대시보드 '콘텐츠 관리'에서 채널별 '게시일' 수동 입력
```

역할 분담: **AI = 생성·기록 자동화 / 작업자(사람) = 검수 + 게시 + 게시일 기록**

## 2. 트리거 명령

| 작업자 입력 | AI 행동 |
|---|---|
| `주제는 ○○○. 내용은 ○○○. 컨텐츠 생성 시작.` | STEP 1 실행 (폴더 + 마스터 원고 + DB '대기' 등록) |
| `반려: {사유}` | 01_main.html 수정 → DB '반려'→수정 후 다시 '대기' 보고 |
| `발행 승인` | STEP 2 실행 (승인·발행일 기록 + 파생 5종 생성 + 플래그 기록) |

## 3. 명명 규칙

- **콘텐츠 ID**: `C{YY}{MM}-{NN}` — 예: 2026년 7월 첫 콘텐츠 = `C2607-01`
  (순번은 해당 월 기존 폴더 수 + 1. `contents/` 폴더 목록으로 확인)
- **폴더**: `contents/C2607-01_{한글 짧은 슬러그}/` — 예: `contents/C2607-01_베어링파손원인/`
- **파일 (생성 순서 고정)**:

| 파일명 | 채널 | 분량 |
|---|---|---|
| `01_main.html` | 마스터 원고 (내부 검수·홈페이지 완전판 겸용) | 2,000자 내외 |
| `02_blog.html` | 네이버/공식 블로그 티저 | 800~1,000자 |
| `03_linkedin.html` | LinkedIn 게시물 | 300자 이내 |
| `04_remember.html` | 리멤버 게시물 | 300자 이내 |
| `05_newsletter.html` | 이메일 뉴스레터 | 본문 300~400자 |
| `06_cardnews_slides.html` | 카드뉴스 1080px 슬라이드 (캡처용) | 슬라이드 5~6장 |
| `07_cardnews_email.html` | 카드뉴스 이메일 발송본 | 카드 3~4개 |

## 4. 작업 절차

### STEP 1 — 메인 콘텐츠 생성 (트리거: "컨텐츠 생성 시작")
1. `contents/` 폴더 목록으로 이번 달 다음 순번 ID 결정.
2. 폴더 생성 → `contents/_templates/01_main_template.html`을 복사해 `01_main.html` 작성.
   모든 `{{ }}` 토큰을 실제 내용으로 채우고 상단 지침 주석은 삭제.
3. 테마 분류: **해결사**(현장 트러블슈팅 사례) / **전문가**(기술 세부 사양) /
   **관리자**(설비 관리 가이드) / **파트너**(전시회·컨퍼런스 등 동반자 가치) 중 택 1.
4. Supabase 기록:
   `python3 scripts/content_db.py register --id C2607-01 --title "제목" --theme 해결사 --folder C2607-01_슬러그`
5. 작업자에게 파일을 제시하고 검수 요청. **여기서 멈춘다. 파생 콘텐츠를 미리 만들지 않는다.**

### STEP 2 — 파생 콘텐츠 생성 (트리거: "발행 승인")
1. `python3 scripts/content_db.py approve --id C2607-01` (검수 '승인' + 발행일 자동 기록)
2. 02→03→04→05→06→07 순서로 템플릿 기반 생성. **모든 파생물은 01_main.html 내용에서만 변환**
   (새로운 사실·수치를 추가하지 않는다).
3. `python3 scripts/content_db.py flag --id C2607-01 --files blog,linkedin,remember,newsletter,cardnews`
4. 생성된 파일 전체를 작업자에게 제시. 채널별 **게시일은 작업자가 대시보드에서 수동 입력**
   — AI가 게시일을 기록하지 않는다.

### 네트워크 폴백
`scripts/content_db.py`는 Supabase 접속 불가 시 자동으로 `contents/_pending_sql/`에
붙여넣기용 SQL 파일을 만든다. 이 경우 작업자에게 "Supabase SQL Editor에서 해당 파일 실행"을 안내한다.

## 5. 콘텐츠 품질 규격 (채널별)

### 공통 금지사항 (전 채널)
- 과장 표현 금지: "세계 최고", "완벽한", "혁신적인" 등
- 수치 근거 없는 효과 주장 금지 / 경쟁사 직접 언급 금지
- 사용자가 제공하지 않은 수치·사례를 지어내지 않는다. 수치가 없으면 `{{수치 확인 필요}}`로 표시하고 검수 시 질문한다.

### 01 마스터 원고
- 독자: 제조업 설비보전·공무·생산관리 담당자 (부장~공장장급)
- 구조: 제목(검색 키워드, 40자) → Hook(3~4문장) → 원인분석(소제목 2~3개) → KLT 솔루션(제품명+수치) → 성과표(Before/After) → CTA
- 톤: 기술 백서 스타일. 신뢰감 있고 객관적. **이모지 금지**

### 02 블로그 티저
- 완전판 수치·도면·세부사양 노출 금지 → 홈페이지 CTA로 유도
- 소제목 H2 적용, 검색 키워드 자연 삽입 (예: 베어링 파손 원인, 자동 윤활 시스템, 그리스 주입 주기)

### 03·04 소셜 (LinkedIn / 리멤버)
- 300자 이내(더보기 없이 노출), 링크 금지(댓글로 공유), 이모지 최대 2개
- 첫 줄 Hook은 숫자 또는 질문형. 제품 스펙 나열 금지, 고객 이익 중심
- LinkedIn=전문가 포지셔닝 톤 / 리멤버=실무 커뮤니티 밀착 톤(경험담형)으로 차별화
- 해시태그: LinkedIn `#KLT #펄사루브 #설비보전 #자동윤활`+주제태그 2 / 리멤버 `#자동주유기 #그리스윤활 #설비보전관리 #B2B기술영업`+주제태그 2

### 05 뉴스레터
- 격식체(하십시오체), 인사말 "…님, 안녕하십니까." 유지, 서명 "KLT 영업팀 ○○○ 배상"
- 개인화 변수는 `[담당자명]` `[산업군]` 형태로 남겨 발송자가 교체

### 06·07 카드뉴스
- 표지 1 + 본문 3~4 + CTA 1. 슬라이드 제목 ≤10자, 본문 ≤40자(2~3줄)
- 06에는 슬라이드마다 `<!-- [시각화 방향] … -->` 주석 필수 (디자이너 전달용)

## 6. 브랜드 스타일 토큰 (전 파일 공통)

| 토큰 | 값 |
|---|---|
| 네이비(주색) | `#2B2E83` / 다크 `#1a1d3d` |
| 옐로(강조) | `#F7B500` |
| 본문 텍스트 | `#212529`, 보조 `#495057`, 흐림 `#8a94a3` |
| 배경 | `#f4f6f9`, 카드 `#f8f9fb`, 선 `#dfe3ea` |
| 폰트 | `'Malgun Gothic','Apple SD Gothic Neo','Segoe UI',sans-serif` |

- **모든 스타일은 인라인(style 속성)으로** 작성 — 블로그·이메일 붙여넣기 호환 목적. `<style>` 블록은 카드뉴스 슬라이드(06)에서만 허용.
- 게시 방법 공통: 파일을 브라우저로 열기 → 전체선택(Ctrl+A) → 복사 → 대상 에디터에 붙여넣기.

## 7. 완료 보고 형식

STEP 완료 시 작업자에게 다음만 간결히 보고한다:
1. 생성된 파일 링크 (present_files 등 파일 카드로 제시)
2. Supabase 기록 결과 (성공 / SQL 폴백 파일 경로)
3. 다음 작업자 행동 1줄 (예: "검수 후 '발행 승인' 또는 '반려: 사유'를 입력해 주세요.")

## 8. 최종 체크리스트 (모든 생성 후 자체 점검)

- [ ] `{{ }}` 토큰이 남아있지 않은가 (미확인 수치 제외)
- [ ] 분량 규격 준수 (03·04는 300자 이내 필수 확인)
- [ ] 파생물이 01_main.html에 없는 내용을 추가하지 않았는가
- [ ] 금지 표현(과장·경쟁사·근거 없는 수치)이 없는가
- [ ] 브랜드 색상·폰트 토큰을 사용했는가
- [ ] Supabase 기록(또는 SQL 폴백)을 수행했는가
- [ ] 게시일은 기록하지 않았는가 (작업자 수동 영역)
